# Whiffle Bakery Backend: My Technical Documentation 🧁

I built this **Whiffle Bakery Backend** as a production-style API using FastAPI. My goal was to create a system that isn't just functional, but also highly scalable, secure, and built with modern asynchronous patterns. I've designed this documentation to explain exactly how I structured the project, why I made specific architectural choices, and how the entire data flow works.

## My Architectural Philosophy and Layered Design

When I started designing this system, I chose a **Service-Oriented Architecture** to keep the codebase clean and modular. I've split the logic into four distinct layers so that the system is easy to maintain and test as it grows. My **API Layer (Routers)**, located in `app/routers/`, is kept intentionally "thin." I only use these files to handle the HTTP-specific concerns, such as defining URL paths, managing query parameters, and ensuring the correct HTTP status codes are returned to the client. This prevents the controllers from becoming bloated with business logic.

The "brain" of my application lives in the **Business Layer (Services)**, found in `app/services/`. I put all my heavy calculations, inventory management, and external service integrations here. By doing this, I ensure that my business logic is independent of the web framework; if I ever decided to switch from FastAPI to something else, my core logic would remain untouched. Supporting this is the **Data Layer (Models)** in `app/models/`, where I define my SQLAlchemy models as the single source of truth for the database schema. Finally, my **Validation Layer (Schemas)** in `app/schemas/` uses Pydantic to strictly enforce data integrity. This ensures that any data entering my system is validated before it ever reaches my business logic, protecting the database from malformed or malicious input.

## How I Engineered the Database Connection

Connecting to a high-performance database in an asynchronous environment like FastAPI required a very specific setup, which I implemented in **[`app/database.py`](file:///f:/projects/Backend/app/database.py)**. I chose to use **SQLAlchemy 2.0** combined with the **`asyncpg`** driver because it's the fastest and most reliable way to communicate with PostgreSQL asynchronously. I store my connection string in a `.env` file to keep my credentials secure and out of my source code.

To manage my database connections efficiently, I created an **Async Engine** with a built-in connection pool. I configured a `pool_size` of 10 to keep a small set of connections constantly open and ready for immediate use, which drastically reduces the latency for the user. I also added a `max_overflow` of 20, which allows the system to temporarily create extra connections if the bakery experience a sudden surge in traffic. To generate my sessions, I use an `async_sessionmaker`. I specifically set `expire_on_commit=False` because, in an asynchronous application, I want to prevent SQLAlchemy from attempting to "lazy-load" data from the database after a transaction has already finished, as that often leads to unexpected errors and performance bottlenecks.

## My Strategy for Authentication and Security

My approach to security is centered around protecting the user's identity without sacrificing performance. I implemented a hybrid system where I use **JWT (JSON Web Tokens)** for stateless authentication, but I never return these tokens in a standard JSON response body. Instead, I store them in an **HttpOnly Cookie**. This is a critical security design because it makes the token invisible to any JavaScript running in the browser, which effectively eliminates the risk of Cross-Site Scripting (XSS) attacks stealing user sessions. I also enforce `Secure=True` to ensure tokens are only transmitted over encrypted HTTPS connections and `SameSite=Lax` to provide a robust defense against Cross-Site Request Forgery (CSRF).

Beyond authentication, I protect my most sensitive operations, like the checkout process, using **Google reCAPTCHA v3**. I built a utility in **[`app/utils/captcha.py`](file:///f:/projects/Backend/app/utils/captcha.py)** that communicates directly with Google's verification servers. When a user attempts to place an order, my backend receives a token which I then verify to get a bot-score. If the score indicates that the user is likely an automated script, I block the transaction immediately. This protects the bakery from automated "brute-force" orders or malicious inventory hoarding. Furthermore, I use **Bcrypt** to hash every password and one-time password (OTP) before they are saved, ensuring that user credentials remain secure even if the database were ever compromised.

## Defining My Data Schema and Models

The structure of my database is defined in the **`app/models/`** directory, where every table inherits from a central `Base` class. In my **User model ([`user.py`](file:///f:/projects/Backend/app/models/user.py))**, I made the decision to use **UUIDs** as primary keys instead of simple integers. This is a deliberate security choice; if I used sequential integers, a malicious actor could easily guess other users' IDs by just changing a number in a URL. UUIDs are random and impossible to guess, providing an extra layer of privacy.

In the **Product model ([`product.py`](file:///f:/projects/Backend/app/models/product.py))**, I use the **`Numeric(10, 2)`** type for all prices. I avoid using floating-point numbers because they are prone to rounding errors that can cause significant financial discrepancies over time. For my **Order model ([`order.py`](file:///f:/projects/Backend/app/models/order.py))**, I implemented a "Snapshot" pattern. When a user places an order, the system captures the exact name and price of the items at that moment and saves them into the `OrderItem` table. This ensures that the user's historical receipt remains accurate even if I change the product's price or name in the catalog later on.

## The Inner Workings of My Service Layer

The most complex logic in my application is housed in the **`app/services/`** folder. My **Order Service**, for instance, manages the entire lifecycle of a transaction. When a checkout is triggered, it doesn't just save a row; it fetches the latest prices from the database to prevent price manipulation from the frontend, verifies that we have enough stock in the inventory, and then updates both the stock levels and the order records in a single, atomic database transaction.

I also built a dedicated **OTP Service** to handle email verification and password resets. To make this as secure as possible, I use the Python `secrets` library, which provides cryptographically strong random numbers that are far superior to standard random generators. These codes are then hashed before being stored, so that even as the system administrator, I cannot see the active codes assigned to users. This service also handles the "invalidation" logic, ensuring that if a user requests a second code, the first one is immediately cancelled to prevent multiple active entry points for the same account.

## How to Get My Project Running and Maintained

Getting the project started is straightforward because I've containerized the infrastructure. I provided a **`docker-compose.yml`** file that allows you to start both the PostgreSQL database and the Redis server with a single `docker compose up -d` command. This ensures that the development environment is identical for everyone. Once the infrastructure is running, I use **Alembic** to manage my database migrations. I treat migrations as "Version Control" for the database schema. Whenever I modify a model file, I run `alembic revision --autogenerate` to generate a script that describes the changes, and then `alembic upgrade head` to apply those changes to the database.

Finally, I've enabled FastAPI's automatic documentation feature. By navigating to the `/docs` endpoint in a browser, you can access the **Swagger UI**. I've configured this so that I can test every single endpoint I've built directly from the browser. It allows me to log in, receive a secure cookie, and then make authorized requests to protected endpoints like the checkout or wishlist, making it an indispensable tool for debugging and verifying my work.
